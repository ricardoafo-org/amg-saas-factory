## The 6 Phases

```
1. SPECIFY  →  2. PLAN  →  3. BUILD  →  4. TEST  →  5. QA/SEC  →  6. LEARN
```

### Phase 1 — Specify (Spec-Driven Development)

Specification is the source of truth — not code, not conversation history.

**What to write before any implementation:**
- Business intent (one paragraph: what problem, for whom, success criteria)
- Acceptance criteria (numbered, testable, unambiguous)
- Constraints (legal, performance, compatibility)
- Out of scope (explicit)

**Format:** Markdown file in `docs/specs/FEAT-XXX-name.md`  
**Gate:** Spec must exist and be approved before any implementer agent touches code.  
**Tool:** Claude Code Plan Mode enforces read-only exploration until approval.

**Why this matters:** Teams using spec-first workflows report 10-15x productivity gains (Anthropic customer data). Skipping the spec is the single largest cause of wasted tokens and rework.

---

### Phase 2 — Plan (Architecture + Approach)

After spec approval, the architect or orchestrator proposes:
- Files to create/modify
- Data model changes
- Agent assignment (which agent does what)
- Dependency order (what must land before what)
- Risk flags (security, compliance, breaking changes)

**Output:** Plan file at `.claude/plans/` (already in use via Plan Mode)  
**Gate:** Orchestrator reviews plan before spawning implementer.  
**Rule:** For changes touching 2+ files, a plan is mandatory. Single-file targeted fixes may skip.

---

### Phase 3 — Build (Implementation)

Only the `implementer` agent writes code.

**Session hygiene (token efficiency):**
- One feature per session — never mix unrelated changes
- Use `/compact` at natural milestones (after completing a component) to preserve context budget
- Keep sessions under 75% context utilization — quality degrades past this
- Delegate investigation/exploration to subagents — tool output is the biggest context drain
- After implementation: run quality gates (type-check → test → lint) before reporting done

**What implementer receives:**
- The spec file path
- The plan file path
- Specific file paths to touch (never "look around")
- Which quality gates to run

**What implementer never does:**
- Reads `.env` or secrets files
- Commits directly to `main`
- Skips quality gates

---

### Phase 4 — Test (Behavior-First Testing)

**The core rule:** Test what the user experiences, not what the code does.

**Pyramid for this project:**

```
E2E (Playwright)           — full user journeys, real browser, real PocketBase
  Integration              — server actions against real PocketBase (not mocks)
    Unit                   — pure functions only (NLP classifier, oil calc, ITV calc)
```

**NLP testing specifically:**
- 50+ Spanish fixture inputs per intent (typos, slang, accent variations, short forms)
- Property tests: "any input containing 'aceite' must never route to a non-oil node"
- Regression corpus: every production misclassification becomes a permanent test case
- Confidence threshold test: inputs < 0.8 confidence must reach Claude fallback

**Property-Based Testing (PBT) — 37.3% better bug detection than example-only:**
- For oil calculator: any km_now > km_last must produce positive km_left
- For ITV: any registration date > 10 years ago must produce annual frequency
- For slot booking: any bookSlot() call must never exceed slot capacity

**Anti-pattern to avoid:**
- Mocking PocketBase in integration tests — we got burned by mock/prod divergence (BUG-003)
- Testing implementation details — test the user-facing behavior

---

### Phase 5 — QA / Security

Run by `qa-engineer` agent before every merge to main.

**Mandatory sequence:**
1. `npm run type-check` — zero exit
2. `npm test` — all pass
3. `npm run lint` — zero errors
4. `npm run flows:validate` — flow valid, no hardcoded values
5. `compliance-reviewer` agent — zero violations (tenant_id, IVA, LOPD order)
6. `security-auditor` agent — IDOR check, PII exposure, auth guards

**Additions from research:**
- Lighthouse CI: fail build if LCP > 2.5s, CLS > 0.1
- axe-core: WCAG 2.1 AA on all pages (legally required in Spain)
- Contract tests: PocketBase schema vs TypeScript types — drift = build failure
- Visual regression: Playwright screenshots on Hero, ServiceGrid, ChatEngine

**Security baseline (27.3% of LLM code has security issues — VeraCode 2025):**
- Every server action reviewed for: SQL/filter injection, missing tenant_id, PII in logs
- LOPDGDD: consent before data, never pre-ticked, policy link present
- Cookies: no non-essential scripts before consent (LSSI-CE)

---

### Phase 6 — Learn (Memory + Retrospective)

After each sprint or major feature:

**Update these artifacts:**
- `CLAUDE.md` — if a new pattern was established or an old one proved wrong
- `.claude/rules/` — if a path-scoped rule emerged
- Memory files (`~/.claude/projects/.../memory/`) — lessons that apply cross-session
- `docs/bugs/` — close resolved bugs, open new ones discovered in QA
- `docs/qa-reports/` — new QA report per session

**What to capture:**
- ✅ Approaches that worked (so they repeat)
- ✅ Approaches that failed and why (so they don't repeat)
- ✅ Every production misclassification (NLP regression corpus)
- ❌ Not: implementation details (those live in code)
- ❌ Not: ephemeral session state

## Working With Claude — Practical Rules

### Context & Token Efficiency

| Rule | Why |
|---|---|
| One feature per session | Context compounds — message 50 costs more than message 5 |
| `/compact` after each component | Preserves budget; summaries cost less than full history |
| Delegate exploration to subagents | Tool output is the biggest context drain in the main session |
| Never paste log files into chat | Use Grep/Read tools or subagents instead |
| Keep CLAUDE.md under 200 lines | Adherence degrades above this threshold |

### When to Use Which Agent

| Task | Agent | Why |
|---|---|---|
| Multi-file feature or bug fix | `implementer` | Only agent with Write/Edit |
| Schema design, API contracts | `architect` | No code, structured analysis |
| Pre-merge compliance check | `compliance-reviewer` | Deterministic grep checks |
| Auth, PII, IDOR review | `security-auditor` | LOPDGDD + IDOR focus |
| Before every merge to main | `qa-engineer` | Bash access, files bugs |
| Codebase exploration | `Explore` subagent | Protects main context |

### When to Use Plan Mode

- ✅ Uncertain approach
- ✅ Changes touching 2+ files
- ✅ New collection / schema change
- ✅ Security-sensitive code (auth, consent, payments)
- ❌ Single-file targeted fix (overhead not worth it)
- ❌ Lint/type fixes with known solution

### Session Hygiene

```
Start session  → state the single task clearly
Mid-session    → /compact after completing each component
End session    → quality gates pass → commit → update docs
New session    → fresh context, don't carry previous session's context manually
```

## Prompt Engineering Patterns

**XML tags** — use to unambiguously separate prompt components when mixing instructions, context, and examples. Names must be semantically meaningful. Combine with chain-of-thought (`<thinking>` + `<answer>`) for structured high-stakes prompts.

**Adaptive thinking** (Claude 4.6+) — more reliable than extended thinking with manual budgets. Claude auto-calibrates how much to think based on request complexity. Use this instead of `extended_thinking` flags for our NLP fallback and complex routing decisions.

**Prompt structure** — put stable context first, variable request last. This is also the optimal layout for prompt caching.

**Prompt caching for API calls** — structure `resolveWithClaude()` in `src/actions/nlp.ts` so the system prompt (business context, option schema) is the stable prefix. Cached tokens cost $0.30/M vs $3.00/M uncached — 10x reduction. Add `cache_control` to the system message block.

## Context Poisoning — Prevention Rules

**Compact at 60%, not 90%** — quality degrades past 75% context utilization. `/compact` early leaves recovery room.

**Compaction preservation instructions** — add to CLAUDE.md: "When compacting, always preserve: tenant isolation rules, LOPDGDD consent order, active bug IDs, and current sprint scope."

**Audit memory files** — first 200 lines of memory files load as high-authority system rules. Review before using in new sessions, especially after merging branches or cloning.

**Never paste log files into chat** — use Grep/Read tools or delegate to subagents. Raw log output is the biggest context drain.

## Agent Patterns (community-validated)

**Builder-Validator chain** — for compliance-sensitive code: one subagent builds, a second validates for cross-tenant leakage, missing consent, hardcoded values. More reliable than single-agent self-review.

**Ralph Loop** — implementer runs tests, checks output against acceptance criteria, fixes and retests in a loop. Include "run the tests and fix any failures" explicitly in the implementer prompt — Claude performs significantly better when it can verify its own work.

**3 Amigos + Critic** (for complex problems) — three parallel exploration agents attempt independently, a critic agent evaluates all three, produces final answer. Use for schema design, security architecture, or API contract decisions.

**Phase-wise gated planning** — for large features: make a phased plan with tests per phase (unit → integration → E2E), then spin up a second Claude as staff engineer to review the plan before implementation starts.

**Domain routing in CLAUDE.md** — define which agent handles which component type. Claude applies routing automatically without being reminded each session.

## Skills vs MCP

- **MCP**: connectivity to external tools/data (PocketBase, Resend, Google Maps)
- **Skills** (`.claude/skills/`): procedural knowledge — how to use that connectivity correctly (LOPDGDD flow, ITV calculation rules, tenant isolation pattern)
- Each skill costs ~100 tokens to scan, <5K tokens when activated — load only what's relevant
